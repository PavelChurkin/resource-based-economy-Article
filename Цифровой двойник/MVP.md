# MVP «Цифровой двойник России»

## Цель MVP

Минимально жизнеспособная версия должна доказать, что «Цифровой двойник России» может объединить разрозненные данные о территории в практический инструмент планирования.

MVP не обязан сразу покрывать всю страну и все закрытые контуры. Его задача — на 2-3 пилотных регионах показать, что система помогает быстрее отвечать на вопросы:

- какие ресурсы, мощности, кадры и инфраструктурные ограничения есть на территории;
- где возникают дефициты, риски и узкие места;
- какие сценарии развития выглядят предпочтительными;
- какие данные уже доступны, а какие требуют межведомственного соглашения или оцифровки.

## Границы MVP

### Входит в MVP

| Блок | Минимальный состав |
|------|--------------------|
| Пилотная территория | 2-3 региона с разным профилем: промышленный, аграрный/ресурсный, транспортный или городской |
| Данные | Открытые и разрешённые ведомственные данные: пространственные слои, статистика, недра, транспорт, ЧС, вакансии, образование, предприятия в агрегированном виде |
| Интерфейс | Веб-карта/3D-визуализация, фильтры по слоям, карточки объектов, аналитические панели |
| Аналитика | Региональный паспорт, поиск дефицитов, простые прогнозы и сравнение сценариев |
| Доступ | Публичный контур с агрегированными данными и ведомственный контур для пилотных участников |
| API | Чтение разрешённых слоёв и выгрузка отчётов |
| Безопасность | Ролевой доступ, журналирование, обезличивание, исключение чувствительных координат |
| Документация | Методика подключения слоя, паспорт набора данных, инструкции пользователя |

### Не входит в MVP

- федеральное покрытие всех субъектов РФ;
- отображение оборонных, специальных и чувствительных логистических данных;
- персональные профили граждан;
- полная автоматизация принятия решений;
- замена существующих государственных информационных систем;
- обязательное подключение всех ведомств и предприятий;
- точное реальновременное моделирование всех физических процессов.

## Пользовательские сценарии

1. **Региональный паспорт.** Аналитик выбирает регион и получает карту ресурсов, производственных мощностей, транспортных узлов, вакансий, образовательных организаций и рисков.
2. **Проверка инфраструктурного проекта.** Пользователь добавляет сценарий строительства объекта и видит влияние на дороги, кадры, энергетику, экологические ограничения и логистику.
3. **Кадровый дефицит.** Служба занятости видит, каких специалистов не хватает в регионе, какие образовательные программы можно использовать и где есть резерв переобучения.
4. **Логистический риск.** Оператор проверяет зависимость территории от одного маршрута, склада или узла и получает варианты резервирования на агрегированном уровне.
5. **ЧС и восстановление.** Ситуационный центр видит доступные ресурсы, безопасные маршруты, медицинские и коммунальные объекты без раскрытия чувствительных данных публичному контуру.
6. **Инвестиционный выбор площадки.** Бизнес или региональный орган сравнивает площадки по инфраструктуре, кадрам, ресурсам, ограничениям и расстоянию до рынков.

## Минимальные функциональные требования

### 1. Сбор и нормализация данных

- загрузка GeoJSON, CSV, API и табличных выгрузок;
- приведение адресов, координат, классификаторов и временных периодов к единому формату;
- хранение источника, даты обновления, лицензии, качества и ответственного владельца данных;
- базовые проверки: дубликаты, пустые координаты, устаревшие значения, конфликтующие записи.

### 2. Геопространственная модель

- карта регионов с включаемыми слоями;
- карточки объектов и агрегированные показатели по району/муниципалитету;
- тепловые карты дефицитов и рисков;
- поиск по объекту, отрасли, территории и показателю;
- экспорт карты или отчёта в PDF/Markdown.

### 3. Аналитические панели

- ресурсы и природные ограничения;
- производство и инфраструктура;
- транспорт и логистика;
- трудовой потенциал, вакансии и обучение;
- экологические и чрезвычайные риски;
- сравнение сценариев «как есть» и «после проекта».

### 4. Сценарное моделирование

Для MVP достаточно простых моделей:

- изменение доступности объекта при отключении одного маршрута;
- оценка кадрового дефицита при запуске нового производства;
- сравнение площадок по расстоянию до ресурсов, кадров и транспортных узлов;
- оценка риска ЧС по историческим и географическим признакам;
- расчёт базовых показателей нагрузки на инфраструктуру.

### 5. Контуры доступа

| Контур | Данные | Пользователи |
|--------|--------|--------------|
| Открытый | Агрегированные и обезличенные показатели | Граждане, бизнес, наука |
| Ведомственный | Детализированные гражданские данные пилота | Органы власти и участники проекта |
| Инфраструктурный | Ограниченные данные по устойчивости объектов | Уполномоченные операторы |
| Тестовый | Синтетические данные и демо-сценарии | Разработчики, студенты, эксперты |

Закрытые и специальные контуры в MVP проектируются методически, но не наполняются чувствительными данными.

## Данные для пилота

Минимальный набор источников:

- НСПД и региональные геослои: границы, земельные участки, объекты инфраструктуры;
- ЕМИСС/Росстат: демография, занятость, отрасли, региональная статистика;
- ФГИС ЕФГИ и открытые материалы Роснедр: недра и минерально-сырьевая база в допустимом объёме;
- OpenStreetMap: дороги, транспортные узлы, базовая инфраструктура;
- открытые данные регионов и муниципалитетов: объекты образования, медицины, коммунальной инфраструктуры;
- службы занятости и образовательные организации: вакансии, программы обучения, агрегированные компетенции;
- МЧС/Росгидромет/региональные источники: природные риски, паводки, пожары, погодные ограничения в разрешённом виде.

## Архитектура MVP

```
Источники данных
      ↓
ETL/ELT-пайплайн проверки и нормализации
      ↓
PostGIS + объектное хранилище + каталог метаданных
      ↓
API доступа и сервис расчёта сценариев
      ↓
Веб-карта, 3D-визуализация, аналитические панели, отчёты
      ↓
Пользователи: регион, ведомства, бизнес, наука, граждане
```

Рекомендуемый технологический контур для MVP:

- **PostgreSQL/PostGIS** — пространственная база данных;
- **объектное хранилище** — архивы, снимки, отчёты, выгрузки;
- **Python/FastAPI** — API, обработка данных, расчёт сценариев;
- **React/Vue + MapLibre/CesiumJS** — веб-интерфейс и карта/3D-визуализация;
- **Apache Airflow или аналоги** — расписание загрузок;
- **Keycloak или отечественный аналог** — роли и доступ;
- **журнал аудита** — запись действий пользователей и изменений данных.

## Этапы MVP

| Этап | Срок | Результат |
|------|------|-----------|
| 1. Концепция и выбор регионов | 1-2 месяца | Утверждены пилотные регионы, роли, данные, риски и метрики |
| 2. Аудит источников | 2-3 месяца | Каталог доступных данных, gaps-анализ, соглашения на обмен |
| 3. Прототип карты | 3-4 месяца | Первые слои, поиск, карточки объектов, базовые панели |
| 4. Аналитика и сценарии | 4-6 месяцев | Региональные паспорта, 3-5 сценарных моделей |
| 5. Закрытый пилот | 2-3 месяца | Тестирование с ведомствами и регионами, корректировка данных |
| 6. Публичный демонстратор | 1 месяц | Открытый контур с агрегированными данными и описанием результатов |

Реалистичный срок MVP: 9-12 месяцев после утверждения регионов и доступа к данным.

## Критерии готовности

MVP считается готовым, если выполнены условия:

1. Подключены 2-3 пилотных региона.
2. В системе есть не менее 10 ключевых слоёв данных.
3. Для каждого слоя указаны источник, дата обновления, качество и режим доступа.
4. Пользователь может собрать региональный паспорт без ручной подготовки презентации.
5. Реализованы минимум 3 сценария: логистика, кадры, инфраструктурный/ЧС-риск.
6. Есть открытый контур без персональных и чувствительных данных.
7. Есть ведомственный контур с ролевым доступом и журналированием.
8. Подготовлена методика подключения нового слоя данных.
9. Подготовлен отчёт о данных, которых не хватает для масштабирования.
10. Пилотные пользователи подтвердили практическую пользу на реальных задачах.

## Метрики эффективности

| Метрика | Целевое значение для MVP |
|---------|--------------------------|
| Время подготовки регионального паспорта | Сокращение минимум в 3 раза по сравнению с ручной сборкой |
| Количество подключённых слоёв | 10-15 на регион |
| Доля слоёв с паспортом качества | 100% |
| Количество сценариев | Не менее 3 рабочих сценариев |
| Обновляемость данных | Не реже 1 раза в месяц для статичных слоёв; чаще для динамических, если источник доступен |
| Время отклика карты | До 3 секунд для типовых фильтров |
| Ошибки доступа | 0 критических нарушений разграничения прав |
| Пользовательская оценка | Не ниже 4 из 5 у пилотных аналитиков |

## Риски MVP и меры снижения

| Риск | Мера |
|------|------|
| Недоступность ведомственных данных | Начать с открытых данных, параллельно готовить соглашения |
| Низкое качество координат и классификаторов | Ввести паспорт качества и отдельный слой проблемных данных |
| Перегрузка проекта требованиями | Жёстко отделять MVP от федеральной целевой модели |
| Риски раскрытия чувствительных данных | Использовать агрегирование, ролевой доступ, аудит и исключение закрытых слоёв из MVP |
| Зависимость от одного поставщика | Опора на открытые форматы, переносимые компоненты и документированные API |
| Недоверие пользователей | Показывать происхождение данных, ограничения и методику расчётов |

## Результаты после MVP

После завершения MVP должны появиться:

- демонстрационная версия платформы для пилотных регионов;
- каталог подключённых и недостающих данных;
- методика безопасного подключения новых источников;
- набор типовых сценариев для регионального планирования;
- расчёт стоимости масштабирования на федеральный уровень;
- предложения по межведомственной модели управления;
- требования к промышленной версии, включая отказоустойчивость, импортозамещение и защиту информации.
